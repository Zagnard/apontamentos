Nomeação de ficheiros

Seja fácil ou difícil de crer, boa escrita HTML, assim como codificação em outras linguagens, começa com uma denominação apropriada segundo regras básicas, assim como intuititiva. Os nomes dos ficheiros são parte da URL do site, de modo que a importância destes não deve ser descurada.

Em adição, apesar de alguns dos pontos explicados nesta página serem exclusivos a HTML e possivelmente XHTML, a maioria são aplicáveis para qualquer linguagem, seja esta de markup ou programação.

Carateres usados

O ideal é nomear os diferentes ficheiros com carateres alfanuméricos (números e letras) e traços (-) ou underscores (_), embora este último seja preterido.

Existem vários carateres que não devem ser usados quando a nomear documentos html, incluindo elementos semânticos de html e outras linguagens (como por exemplo os sinais de menor (<) e maior (>) para HTML, onde respetivamente representama abertura e o fecho de uma tag) ou de outras linguagens, carateres que não são devidamente processados por servidores web e até mesmo carateres que afetam a pontuação do site pelos motores de busca.

Vamos analisar e explicitar estes diferentes pontos individualmente.

Letras maiúsculas ou minúsculas?

É altamente aconselhável usar apenas letras minúsculas para nomes dos ficheiros.

Um primeiro ponto a ter em conta é o facto de ser relativamente mais fácil (menos probabilidade de erro, menos teclas premidas, etc). Isto reduz também a probabilidade de erro quando a fazer alguma ligação e o link não funcionar.

Um outro ponto a que se deve prestar atenção é maximizar a compatibilidade entre diferentes sistemas e servidores. Enquanto um sistema microsoft não distingue entre uma letra maiúscula e minúscula, um sistema linux faz esta distinção, devido a organizar os seus ficheiros com base no código do caracter, que naturalmente, é diferente entre letras minúsculas e maiúsculas.

Vários servidores web são programados em linux, e quando a transferir ficheiros de um desses servidores para microsoft, se houver ficheiros com o mesmo nome, mas capitalização diferente para distinção no servidor, o windows irá substituir um desses ficheiros por não fazer distinção baseada na capitalização. Há também servidores programados em outras linguagens (ex: Apache), que não suportam capitalizações.

Como tal, para pessoas que estão habituadas a linguagens de programação, fazer uso de camelCase (palavras juntas, mas com a letra capitalizada a partir da segunda ex: thisIsACoolWebsite) não é aconselhado.

Espaços, traços ou underscores?

Nós estamos habituados a separar as palavras com espaços. Infelizmente, em termos de programação, isto levanta alguns problemas.

No geral, em linguagens de programação e de markup, o espaço indica o fim de um conjunto de carateres (string), de modo que quando incluímos links que têm espaços no nome, estes podem ser mal interpretados e gerar problemas.

O nome do ficheiro faz também parte da URL, mas o servidor tem tendência a codificar o espaço com %20, gerando uma URL menos apelativa. Como as pessoas podem consultar a URL por pistas para os conteúdos da página, isto pode reduzir confiança no website.

A outra desvantagem de usar espaços nos nomes dos ficheiros pode levar a que estes sejam corrompidos quando são gravados, reencaminhados ou ligados a, através de uma hiperligação.

Underscores, contudo, têm algumas desvantages. Uma delas é o facto de não poderem, ou dificilmente poderem ser vistos em hiperligações que não tenham o sublinhado omitido, em comparação com traços.

Outra desvantagem, é quando se tem o site na internet e se espera que hajam utilizadores que a encontrem através de um motor de busca.

Para um motor de busca, um traço tem a conotação de espaçamento, de modo que aqui-falamos-de-animais continua a ser aqui-falamos-de-animais. Contudo, o underscore tem uma conotação de concatenação (palavra cara para "junção" ou "ligação"), de modo que aqui_falamo_de_animais passa a ser aquifalamosdeanimais.

Tendo em conta que a URL de um website inclui o nome do ficheiro html ou aberto, normalmente no fim, o motor de busca eventualmente irá ler essa parte da URL. Se tiver traços, consegue fazer distinção entre as diferentes palavras e já anota algumas palavras chave desse website e dessa página. Se tiver underscores, tem-se essencialmente uma amalgamação de palavras sem sentido para o motor de busca e de difícil leitura para utilizadores.

Como tal, pode-se concluir que o melhor carater para separar palavras quando a nomear ficheiros html, e vários outros, é o traço.

Escolher um bom nome

Agora que entendemos que carateres podemos ou não usar quando a nomear um ficheiro html, há que ter algumas considerações relativamente à constituição dos nomes.

Ficheiros em geral

Isto aplica-se à designação de qualquer ficheiro que façamos uso em programação: o nome deve ser descritivo do conteúdo, em vez de uma linha de carateres aparetenemten aleatórios e idealmente não deve ser demasiado comprido, para ser visto por completo no hover da hiperligação e para ser mais fácil de recordar (devo admitir que falho redondamente neste ponto).

É também aconselhável não começar o nome do ficheiro com um número, uma vez que números como primeiro carater de uma string tem uma contoção específica em algumas linguagens e/ou programas que podem ser usados para interagir com a página em questão, levando a resultados indesejados.

Nome da página principal

Independentemente do nome do website, há que ter em conta que já existe um nome definido para a página princpal num servidor web, sendo este "index".

Quando um servidor web tenta abrir um website, precisa de escolher uma das páginas do website para abrir de entre das várias que estão na pasta do website. Como tal, o servidor web começa com nomedowebsite.pt/ , mas depois adiciona automaticamente o nome padrão usado para a página principal, sendo este index. Assim, embora a ligação nos apareça algo do tipo www.nomedowebsite.pt/ , a página aberta é www.nomedowebsite.pt/index.

Este não é o método ideal, e só funciona quando se tem o site num servidor, mas pode-se usar este padrão como uma pequena medida de proteção. Como o servidor procura logo a página index como a página a ser aberta, pode-se criar uma página index, mesmo sem conteúdo, dentro de cada uma das subpastas principais. Quando algum possível cracker tente aceder a essa pasta através da URL, para ver a estrutura da pasta, será automaticamente redirecionado para a página index lá presente, mantendo os conteúdos da página ocultos. Extensão do ficheiro

Finalmente agora que já se sabem os carateres permitidos no nome de um ficheiro html e como escolher um bom nome, deixo uma consideração final relativamente à extensão do mesmo.

A extensão de um ficheiro consiste na parte do nome do ficheiro que está a seguir ao ponto. Essa parte determina o tipo de ficheiro que é, e como este deve iniciar. Caso não consiga ver essa parte do nome, isso pode ser ativado nas opções do sistema operativo (windows, mac, etc).

No caso de ficheiros html, estes devem terminar com a extensão .htm ou html, sendo ambas equivalentes uma à outra, mas é aconselhável usar apenas uma para evitar confusões. Quando se estiver a trabalhar com uma equipa, é aconselhável definir não só esta extensão, mas também a estrutura dos nomes dos ficheiros e das pastas, assim como a estrutura das pastas em si.

Para outros ficheiros comumente usados em desenvolvimento web: os ficheiros css têm extensão .css, os javascript em .js e os ficheiros de imagem têm normalmente como extensão .png e/ou .jpg.

Em suma

Uma boa nomeação não serve meramente para nos ajudar a identificar ficheiros, mas também a termos uma boa organização e menos probabilidade de erro tanto por nós, como pelos diferentes programas que usamos. Em suma deve-se:

  • Escrever apenas em letras minúsculas e números, a começar o nome com uma letra e a usar traços para espaços;
  • Criar nomes relativamente curtos, mas explicativos;
  • Nomear a página principal index.htm ou index.html;
